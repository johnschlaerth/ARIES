ARIES SYSTEM DESCRIPTION
Automated Robotic Integrated Effects System
LHXelerator Disruption Challenge #4 Foundation Document
Working Version: Quest 2 / MVP Definition

1. EXECUTIVE SUMMARY

ARIES, the Automated Robotic Integrated Effects System, is a near-term robotic ground effects concept that combines venture-partner vehicle autonomy with L3Harris communications, spectrum dominance, ISR, counter-UAS, C2, and modular effects technologies.

The core concept is simple: take reasonably low-cost vehicles, retrofit them into teleoperated or autonomy-ready robotic ground platforms using Polymath Robotics, then integrate L3Harris mission systems and selected payload modules to create attritable ground nodes that can sense, relay, jam, defend, deliver effects, and support human operators in dangerous terrain without exposing personnel.

ARIES is not a new vehicle prime program. It is an integration and orchestration layer. The value proposition is that L3Harris can use venture-partner mobility to extend existing L3Harris capabilities into tactical ground positions that would otherwise require exposed personnel, expensive manned vehicles, or slow bespoke platform development.

Primary venture partner:
Polymath Robotics

Secondary payload partner:
Allen Control Systems

Primary L3Harris focus:
Communications and Spectrum Dominance

Secondary L3Harris focus:
Space and Mission Systems / ISR / C2 / launched effects / Missile Solutions roadmap

Primary demonstration target:
A local software MVP that simulates AI-assisted tasking of robotic ground effects nodes in a synthetic battlespace. The software shows target classification, priority scoring, terrain-aware movement, effect assignment, tactical visualization, and mission outcome reporting. The MVP does not connect to real hardware, does not control weapons, and does not model classified capabilities. It demonstrates the decision-support and orchestration logic that would assist teleoperators or future supervised autonomy.

2. THE CORE THESIS

Modern battlefields increasingly require distributed, mobile, attritable, and rapidly reconfigurable systems. Ukraine has shown that low-cost robotic and unmanned systems can generate outsized tactical effects, especially when paired with sensing, networking, electronic warfare, and precision effects.

L3Harris already has the critical mission systems:
- Tactical radios
- Resilient mesh networking
- Anti-jam waveforms
- AI-enabled spectrum sensing
- EO/IR payloads
- Full-motion video relays
- Counter-UAS kits
- C2 software
- Launched effects
- Missile Solutions / propulsion expertise

What L3Harris does not need to become is a vehicle prime. ARIES solves that by using Polymath Robotics as the autonomy and vehicle retrofit layer. Polymath provides the robotic ground mobility substrate. L3Harris provides the mission systems, communications backbone, effects integration, and AI orchestration layer.

The result is a low-cost, vehicle-agnostic, modular robotic effects node that can be fielded faster than a purpose-built unmanned ground combat vehicle and can be adapted to multiple customer missions.

3. THE ONE-SENTENCE VERSION

ARIES turns low-cost vehicle platforms into teleoperated or autonomy-ready robotic ground effects nodes by combining Polymath Robotics vehicle automation with L3Harris tactical communications, spectrum dominance, ISR, C-UAS, C2, and modular payload capabilities.

4. STRONG REVIEWER-FACING VERSION

ARIES is a near-term, modular robotic ground effects architecture that gives L3Harris a rapid path to field attritable, vehicle-agnostic tactical nodes for contested communications, electronic warfare, ISR relay, counter-UAS, and future launched effects. By using Polymath Robotics to retrofit existing vehicles and L3Harris systems to provide resilient communications, sensing, C2, and payload integration, ARIES allows the company to extend existing mission systems into dangerous tactical positions while reducing risk to personnel and avoiding the cost and schedule burden of becoming a vehicle prime.

5. PROBLEM STATEMENT

Ground forces increasingly need to project sensing, communications, electronic warfare, counter-UAS, and precision effects into high-risk zones. Today, that often requires one or more undesirable choices:
- Expose personnel to danger
- Use expensive tactical vehicles for dangerous forward missions
- Deploy dismounted teams into areas with drone, artillery, mine, EW, and ambush risk
- Build bespoke robotic vehicles with long development timelines
- Treat radios, sensors, jammers, turrets, and launched effects as isolated systems rather than coordinated distributed nodes

At the same time, the tactical environment is shifting:
- Drones and small unmanned systems are becoming cheaper, more numerous, and harder to detect early
- RF environments are increasingly contested and congested
- Static defensive systems are vulnerable
- Manned logistics and forward positioning are expensive and dangerous
- Customers need faster fielding paths, not exquisite bespoke platforms

ARIES addresses this problem by creating an attritable, modular ground platform architecture that can carry, position, and coordinate L3Harris mission effects without requiring a human operator to occupy the danger zone.

12. REVIEWER-FACING TECHNICAL NARRATIVE

The technical core of ARIES is not the vehicle. The technical core is distributed effect orchestration over a resilient tactical network.

A fleet of robotic ground nodes is treated as a dynamic graph:
- Each node has a position
- Each node has a velocity
- Each node has a payload type
- Each node has an effect radius
- Each node has a communications state
- Each node has a health state
- Each node has mission utility
- Each threat has a classification
- Each threat has a confidence level
- Each threat has a priority score
- Each threat has a proximity relation to friendly nodes
- Terrain modifies mobility and line-of-sight assumptions

The ARIES battle manager computes global tasking recommendations. Each node also computes local recommended behavior based on proximity, threat level, payload match, terrain, and survivability. The result is a visible prioritization and routing system that can support teleoperators or serve as a future supervised autonomy layer.

16. CONOPS

16.1 Baseline Mission

A small team needs to defend or hold a contested area with reduced personnel exposure. The area includes enemy drones, enemy ground units, possible EW nodes, and unknown contacts. Human operators cannot safely occupy several forward positions.

The ARIES system deploys several robotic ground vehicles:
- One communications relay node
- One EW / spectrum node
- One ISR node
- One C-UAS node
- One expendable close-effect node

The vehicles are teleoperated or supervised by operators. The ARIES mission software ingests entity data, image classifications, simulated sensor reports, terrain data, and vehicle status. It then recommends which vehicles should move, which threats should be prioritized, and which effect category should be assigned.

16.2 Step-by-Step Operational Flow

Step 1:
Operator loads mission configuration.

Step 2:
ARIES displays synthetic battlefield map with terrain contours, friendly robotic nodes, enemy entities, unknown contacts, and mission objective.

Step 3:
Image classification module processes photos from an image folder and assigns each image:
- entity name
- category
- allegiance
- threat level 1-10
- description
- confidence
- rationale
- recommended simulation class

Step 4:
Simulation initializes friendly and enemy entities.

Step 5:
Central battle manager computes global priorities:
- highest threat
- closest threat to defended asset
- highest mobility threat
- threat to network node
- payload match score
- terrain accessibility
- friendly survivability

Step 6:
Each friendly robotic vehicle computes local priority:
- closest relevant threat
- payload compatibility
- path cost
- risk of loss
- support value to other friendly units

Step 7:
Simulation advances in fixed timesteps.

Step 8:
Friendly nodes move according to terrain-aware route recommendations.

Step 9:
Enemy nodes move according to simple behavior models:
- drones fly straight or near-straight toward objective / defended asset
- ground threats move through lower-cost terrain
- EW threats remain mostly stationary or move slowly
- unknowns may become classified after ISR exposure

Step 10:
Effects are resolved abstractly:
- EW suppresses
- C-UAS neutralizes drones
- effect vehicles neutralize ground threats
- relay vehicles improve command/network range
- ISR vehicles improve classification confidence

Step 11:
Killed / disabled entities remain on-screen as X marks.

Step 12:
Mission report is generated:
- threats neutralized
- friendlies lost
- objective status
- network coverage score
- average response time
- recommended assignments
- final entity table
- event log

17. MVP DEMONSTRATION TARGET

The Quest 2 MVP should be intentionally narrow:

“Demonstrate a local Python simulation in which AI-classified entities are placed into a synthetic contour-map battlespace, scored by a central ARIES battle manager and local vehicle-level decision logic, then visualized in a 1980s-style tactical display as robotic ground nodes move, prioritize threats, apply abstract effects, and generate a mission outcome report.”

This is the demonstration target because it directly answers prior feedback:
- It narrows the broad concept into a specific prototype
- It focuses on the AI planning layer
- It shows real software working
- It avoids dependency on hardware procurement
- It demonstrates incremental transition path
- It makes the concept technically legible to senior fellows

20. RISKS AND MITIGATIONS

Risk 1: Scope is too broad
Mitigation:
Define Quest 2 as a software-only mission-effect simulator. Treat physical payloads as roadmap modules, not MVP dependencies.

Risk 2: Reviewers think capability already exists
Mitigation:
Position ARIES as the integration/orchestration layer across existing capabilities, not as a claim that each subsystem is novel.

Risk 3: Vehicle autonomy is outside L3Harris core
Mitigation:
Make Polymath Robotics the vehicle autonomy layer. L3Harris stays focused on communications, spectrum, ISR, C2, and effects integration.

Risk 4: Weapon autonomy creates review friction
Mitigation:
Frame MVP as teleoperator decision support and abstract simulation. Use human-supervised language for real-world kinetic effects. Do not model real fire-control or payload construction.

Risk 5: No sponsor / endorser
Mitigation:
Identify a Communications and Spectrum Dominance technical leader, fellow, or senior engineer who can endorse the value of robotic ground networking / distributed spectrum nodes.

Risk 6: Export control
Mitigation:
Keep Quest 2 material based on public product information, synthetic simulation, abstract effects, no classified performance values, no controlled technical data, and no implementation detail for real weapons or jamming.

Risk 7: Demo reliability
Mitigation:
Build a deterministic local simulation with mock mode. API classification should be optional, cached, and never required for the live demo.
